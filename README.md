# CareerOS / Career CRM

Локальный AI-ассистент для поиска работы. Собирает вакансии с hh.ru, анализирует их с помощью AI, помогает создавать сопроводительные письма и отслеживать отклики.

**Никакой автоматической отправки откликов. Все отклики — только вручную пользователем.**

---

## Быстрый старт

```bash
npm install
npx prisma migrate dev
$env:DATABASE_URL='file:./dev.db'; npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000) и пройдите онбординг: добавьте резюме и настройте AI.

## Воронка вакансий

Каждая вакансия проходит через следующие этапы:

```
found → ai_recommended → ready_to_apply → applied → waiting_response
                ↓                                         ↓
          needs_review                         rejected / no_response
                ↓
          skipped / archived
```

| Статус | Русское название | Смысл |
|--------|-----------------|-------|
| `found` | Найдена | Вакансия собрана, анализ ещё не выполнен |
| `ai_recommended` | AI рекомендует | AI считает вакансию подходящей |
| `needs_review` | На проверке | Неоднозначная оценка, нужен взгляд пользователя |
| `ready_to_apply` | Готово к отклику | Есть письмо, можно открывать hh и откликаться |
| `applied` | Отклик отправлен | Пользователь вручную отправил отклик |
| `waiting_response` | Ждём ответ | Отклик отправлен, ожидаем обратную связь |
| `no_response` | Нет ответа | Ответа не поступило |
| `rejected` | Отказ | Получен отказ |
| `skipped` | Пропущена | Пользователь решил пропустить |
| `archived` | Архив | Вакансия убрана из активной очереди |
| `analysis_error` | Ошибка анализа | AI не смог разобрать вакансию |
| `invalid_source` | Невалидный источник | Служебная страница hh, не вакансия |

### Вкладки на странице /vacancies

- **Все** — все актуальные вакансии, кроме архивных, пропущенных и невалидных источников
- **AI рекомендует** — вакансии со статусами `ai_recommended` и `ready_to_apply`
- **На проверке** — статус `needs_review`
- **Готово к отклику** — вакансии с созданным письмом (`ready_to_apply` или `ai_recommended` + письмо)
- **Отклик отправлен** — статусы `applied`, `waiting_response`, `no_response`
- **Без AI-анализа** — вакансии, готовые к запуску анализа
- **Ошибка анализа** — `analysis_error`
- **Невалидные источники** — `invalid_source`
- **Архив** — `archived`, `skipped`

## Как создать и отправить отклик

1. Запустите поиск вакансий (`/search`).
2. Запустите быстрый AI-анализ (кнопка "Быстрый AI-анализ" на `/vacancies`).
3. Откройте вкладку "AI рекомендует" — найдите интересные вакансии.
4. Нажмите "Создать письмо" на карточке вакансии.
5. На вкладке "Готово к отклику" появится вакансия с письмом.
6. Нажмите "Скопировать письмо", откройте вакансию на hh, вставьте письмо и **вручную** отправьте отклик.
7. Вернитесь в CareerOS и нажмите "Отклик отправлен" — вакансия уйдёт в раздел `/applications`.
8. Через 5 дней проверьте ответ от работодателя.

## Компании

Страница `/companies` показывает список компаний из вакансий с краткой статистикой.

**Как создаются карточки компаний:**
- При каждом сборе вакансий с hh.ru система автоматически создаёт или находит компанию по имени и привязывает вакансию к ней.
- При добавлении вакансии вручную компания также создаётся автоматически.
- Дублей не создаётся: если компания с таким именем уже есть, используется существующая.

**Backfill (создание компаний из старых вакансий):**

Если вакансии были собраны до того, как привязка к компаниям заработала, нажмите кнопку **"Создать компании из вакансий"** на странице `/companies`. Система попробует связать вакансии без компании по следующим критериям:
1. Общий `employerUrl` (URL работодателя на hh.ru) — самый надёжный метод.
2. `companyId` из истории взаимодействий (`Interaction`) при создании вакансии.

После backfill компании появятся на странице `/companies`.

## Стиль сопроводительных писем

CareerOS пишет письма в деловом, спокойном, человеческом тоне — без пафоса и без хвастовства.

**Структура письма:**
1. "Здравствуйте."
2. "Рассматриваю вашу вакансию [название]."
3. 2–3 ключевые задачи из описания вакансии.
4. 2–3 факта из резюме, подтверждающие релевантный опыт.
5. "Готов обсудить задачи, условия и формат работы."

**Фильтрация релевантных фактов:**

AI выбирает из `candidate_strengths` только те факты, которые прямо связаны с задачами вакансии (`matched_requirements`, `job_priorities`). Нерелевантные факты передаются writer как `excluded_facts` и не упоминаются в письме. Например:
- KPI и сокращение сроков договоров — только для управленческих/процессных вакансий.
- Досудебное урегулирование — только для претензионно-судебных вакансий.
- Руководство отделом — только если вакансия явно руководящая.

**Предупреждения после генерации:**

После создания или перегенерации письма система автоматически проверяет текст и показывает предупреждения (жёлтый блок), если письмо:
- содержит KPI, не связанный с вакансией;
- упоминает сокращение сроков договоров для нерелевантной позиции;
- использует фразу "сейчас я руковожу..." для не-руководящей вакансии;
- называет прошлого работодателя;
- обещает конкретный формат работы (удалёнка / офис);
- слишком длинное (более 1200 символов);
- не содержит формулировку зарплатных ожиданий, хотя работодатель просит и настройки заполнены.

Предупреждения носят рекомендательный характер — вы всегда можете отредактировать текст вручную.

**Что AI никогда не делает:**
- не упоминает прошлых работодателей и их названия
- не обещает удалёнку, офис или конкретный график без подтверждения пользователя
- не использует пафосные слова ("уникальный", "идеальный", "выдающийся")
- не придумывает опыт и навыки

**Стили перегенерации** (на странице вакансии):
- Спокойное (базовое) — нейтральный деловой стиль
- Короткое — 3–4 предложения
- Акцент на судебную работу
- Акцент на договорную работу
- Акцент на управление

## Редактирование письма перед копированием

На странице вакансии (`/vacancies/[id]`), в блоке "Сопроводительное письмо":

1. Текст письма отображается в редактируемом поле (`textarea`).
2. Отредактируйте текст прямо в поле.
3. Нажмите "Сохранить правки" — правки сохраняются в базе.
4. Нажмите "Скопировать" — скопируйте итоговый текст в буфер обмена.
5. Кнопка "Вернуть AI-версию" появляется при наличии несохранённых изменений.

## Зарплатные ожидания

На странице `/settings/profile` задайте:

- **Минимум / Максимум** — диапазон ожидаемой зарплаты в рублях.
- **Формулировка для письма** — текст, который вставится в сопроводительное письмо, если работодатель просит указать ожидания.
- **После вычета налогов** — флаг, использующийся только как подсказка для себя.

**Как настройки используются:**

| Этап | Что происходит |
|------|----------------|
| AI-анализ | Если зарплата вакансии явно ниже минимума — добавляется red flag "Зарплата ниже ожиданий". Если зарплата не указана — red flag "Зарплата не указана". |
| Признак запроса | AI выставляет флаг `salary_expectations_requested = true`, если вакансия содержит фразы типа "укажите зарплатные ожидания". |
| Письмо | Если флаг выставлен и заполнена "Формулировка для письма" — фраза вставляется в конец письма. Если поле пустое — нейтральная фраза "Готов обсудить задачи, условия и зарплатные ожидания." |
| Предупреждение | После генерации письма, если флаг выставлен и формулировка заполнена, но в тексте нет зарплатной фразы — показывается предупреждение. |

**Сумму AI не придумывает.** При создании / перегенерации письма вручную можно переопределить формулировку для конкретной вакансии.

## "Не подходит" и обучение AI

На любой карточке вакансии и в списке есть кнопка **"Не подходит"**.

При нажатии открывается модальное окно с:
- набором быстрых причин (Не мой профиль, Нет опыта в специализации, Мало денег и т.д.)
- полем для свободного комментария

**После сохранения:**
1. Вакансия переходит в статус `skipped` и скрывается из рекомендаций и списка "AI рекомендует".
2. Создаётся событие `Interaction` ("Вакансия отклонена пользователем").
3. Если указаны причины или комментарий — создаётся **черновое наблюдение** (`LearningObservation` со статусом `draft`).

**Как черновик превращается в правило:**
1. Откройте `/memory` ("Память AI").
2. Черновик появится в блоке "Черновые наблюдения" с предложенным правилом.
3. Нажмите **"Принять как правило"** — статус становится `accepted`.
4. При следующем AI-анализе вакансий правило передаётся в промпт и влияет на оценку.

**Важно:** черновые (`draft`) и отклонённые (`rejected`) наблюдения на анализ не влияют. Влияют только `accepted`.

---

## Draft-наблюдения и принятые правила

**`/memory`** ("Память AI") показывает черновые наблюдения и активные правила.

| Состояние | Влияет на AI? |
|-----------|---------------|
| `draft` | Нет. Ожидает подтверждения пользователя. |
| `accepted` | Да. Передаётся в промпт при анализе каждой новой вакансии. |
| `rejected` | Нет. |

Для каждого черновика доступны действия:
- **Принять как правило** → статус `accepted`, правило начинает влиять на анализ.
- **Изменить** → отредактировать формулировку правила и принять.
- **Отклонить** → статус `rejected`, не используется.
- **Удалить** → удалить из базы.

## Почему AI не рекомендует узкие специализации

AI использует список узких специализаций из `src/lib/narrow-specializations.ts`. Если вакансия требует как **основной профиль** одну из специализаций (патентное право, M&A, банкротство и т.д.), а в резюме этот опыт явно не указан:

- `score` ≤ 45
- `should_apply = "no"`
- Специализация попадает в блок "Узкие требования не найдены в резюме"

Список можно расширить, отредактировав файл `src/lib/narrow-specializations.ts`.

## Как смотреть причины рекомендации

На странице вакансии (`/vacancies/[id]`) в блоке **"AI-разбор"**:

- **Почему подходит** — требования, подтверждённые резюме.
- **Слабые совпадения** — частичное совпадение.
- **Красные флаги** — тревожные сигналы: зарплата ниже ожиданий, нет опыта в профиле, мутный формат и т.д.
- **Неподтверждённые требования** — что вакансия требует, но в резюме не найдено.
- **Почему AI так решил** — блок `resume_match_basis`:
  - что совпало с резюме;
  - что не подтверждено;
  - узкие требования, не найденные в резюме;
  - итоговая причина рекомендации / отказа.

## Безопасность и приватность

Репозиторий **публичный**. В git не должны попадать:

- `.env`, `.env.local` — ключи API
- `prisma/dev.db` — SQLite с вашими данными
- `.next/`, `node_modules/` — сборочные артефакты
- `browser-profile/`, `uploads/`, `logs/` — локальные данные
- `resumes/`, `memory/`, `knowledge/`, `app_data/` — персональные данные

Всё перечисленное исключено в `.gitignore`.
